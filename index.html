First Load Limitations
& Fallback Strategy

Understanding the 5–6s latency challenge on OTP init and how we address it today.

OTP · TUI Team
The Problem

Every order starts with a wait

5–6s
BFF response latency
on first load (all orders)
📦

Root cause

BFF aggregates data from central logistics & upstream services (e.g. TAPI). These upstream dependencies add significant latency on every request.

🗺️

User impact

Map pins and ETA cannot render until BFF responds. Users see no useful info for the first 5–6 seconds after the OTP opens.

⚠️

Additional risk

Beyond latency: if BFF is down or returns an error, there is currently no data source to fall back on.

Current State

What happens when OTP opens today

📱

Host App opens OTP

Brand initialises the Tracking SDK

🔄

SDK calls BFF

BFF fans out to TAPI & upstream logistics

⏳

5–6s wait

User sees empty / loading state during this gap

✅

Data arrives

Map & Order Status render with live data

!

Loading & error states are today fully owned by Brands

High variability across brands — full-screen loaders, shimmer effects, blank screens — with no unified SDK definition.

!

First-load error: fallback to Brand map (if available)

Once TSDK Component fully replaces Brand maps, this fallback disappears. We need a defined strategy before that point.

Options Evaluated

Three approaches considered

1

Client-provided initial data Chosen

Host app passes a small set of initial data at SDK init. SDK renders immediately while BFF fetch runs in parallel.

2

BFF fetches from alternative sources Discarded

BFF integrates with additional upstreams (e.g. OTv2). Doesn't help when BFF itself is down. Adds BE complexity and may not remove the delay (e.g. Glovo sends to OTv2 with 3s+ delay).

3

Loading state only Discarded

Show a spinner until BFF responds. Affects 100% of orders, changes UX in first seconds, provides no useful info, and is especially bad during incidents.

Adopted Solution

Client-provided Initial Data at SDK init

How it works

  • Host app passes Initial Data when initialising the SDK
  • SDK renders map pins & Checkout ETA immediately from this data
  • In parallel, SDK fetches full data from BFF
  • UI updates once BFF response arrives
  • BFF timeout / error (>5 min): keep Initial Data + toast "live updates temporarily unavailable" + retry

Initial Data payload Optional · Strongly recommended

  • pickupLocation — lat, lng, pickupLabel
  • dropoffLocation — lat, lng, dropoffLabel
  • checkoutEta — timestamp

Brands should already have this data at checkout. The set is intentionally minimal to remain stable over time.

Why this is the right trade-off

Key benefits

✓

Best UX — immediate, meaningful UI for every order

Covers the universal 5–6s delay. Users see map pins and ETA instantly instead of a blank screen.

✓

Graceful degradation during BFF incidents

If BFF is down for extended periods, we surface a clear toast and retry — rather than showing nothing.

✓

Limited Brand burden

Only 3 fields, all already available at checkout. Brands opt in; the interface is stable and unlikely to change.

✓

Low long-term risk

Staying strict about what data Brands pass keeps the SDK as the single source of truth once BFF data arrives.

Open Questions & Next Steps

Error States

  • First-load errors with no fallback map available

  • Error message placement — toast, overlay, inside component
  • Partial failures — one component fails, the other works
